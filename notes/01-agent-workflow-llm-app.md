# Agent、Workflow、LLM App 的区别

学习目标：理解三类 AI 应用形态的边界，知道什么时候该用简单 LLM App，什么时候该做 Workflow，什么时候才需要 Agent。

## 一句话区分

- LLM App：用大语言模型增强某个功能，核心是“输入 -> 模型处理 -> 输出”。
- Workflow：把一组明确步骤串起来，核心是“固定流程自动化”。
- Agent：让模型根据目标、环境和反馈动态决定下一步，核心是“自主规划与行动循环”。

## 1. LLM App 是什么

LLM App 指接入大语言模型能力的应用。它通常围绕一个明确功能展开，比如总结文章、改写文案、翻译、生成 SQL、问答助手、代码解释器。

典型结构：

```text
用户输入 -> Prompt 模板 -> LLM -> 结果展示
```

它的特点：

- 流程通常很短，常见是一轮或少量多轮对话。
- 模型主要负责生成、理解、分类、抽取、改写。
- 应用逻辑大多由代码控制，模型只是其中一个能力模块。
- 工具调用不是必须的。
- 失败时通常由用户重新提问，或由程序做简单重试。

例子：

- 输入一段会议记录，输出摘要。
- 输入一句自然语言，生成 SQL。
- 上传一篇文章，生成要点和行动项。
- 在代码编辑器里解释某段代码。

你需要重点理解：

- Prompt 如何组织。
- 输入输出格式如何约束。
- 模型幻觉如何降低。
- 如何把 LLM 当作一个“不稳定但强大的函数”来接入软件。

适合使用 LLM App 的场景：

- 任务边界清楚。
- 不需要复杂工具调用。
- 不需要模型自己规划多步任务。
- 输出质量可以由用户快速判断。

## 2. Workflow 是什么

Workflow 是固定步骤的自动化流程。它可以包含 LLM，也可以不包含 LLM。LLM 在 Workflow 里通常是某个节点，用来做分类、总结、生成、判断或转换。

典型结构：

```text
步骤 1 -> 步骤 2 -> 条件判断 -> 步骤 3 -> 输出
```

例如“每日 AI 简报”：

```text
定时触发
-> 搜索新闻
-> 筛选新闻
-> LLM 总结
-> 生成 Markdown
-> 发送到飞书/邮箱
```

它的特点：

- 流程由开发者提前设计好。
- 每一步的顺序、条件、输入输出比较明确。
- 适合稳定、重复、可预期的业务。
- 可观测性和可测试性通常比 Agent 更好。
- 低代码平台 Coze、Dify、n8n 常用来搭建这类系统。

Workflow 也可以有分支：

```text
如果用户问题是售后 -> 走售后流程
如果用户问题是退款 -> 走退款流程
如果用户问题是技术问题 -> 走技术支持流程
```

但这些分支通常是预先设计好的，不是模型自由探索出来的。

你需要重点理解：

- 节点：每一步做什么。
- 数据流：上一步输出如何成为下一步输入。
- 条件分支：什么时候走哪条路径。
- 错误处理：某一步失败后如何重试、跳过或人工介入。
- 可观测性：如何记录每一步的输入、输出和耗时。

适合使用 Workflow 的场景：

- 业务流程稳定。
- 步骤可以提前枚举。
- 需要可靠、可控、可审计。
- 出错成本较高，不希望模型随意决定流程。

## 3. Agent 是什么

Agent 是能围绕目标自主选择行动的系统。LLM Agent 通常以大语言模型作为“决策核心”，结合工具、记忆、状态和反馈循环完成任务。

典型结构：

```text
目标
-> 观察当前状态
-> 思考下一步
-> 选择工具或行动
-> 执行
-> 获取观察结果
-> 更新状态
-> 继续循环，直到完成或停止
```

这就是常说的 Agent Loop：

```text
Observe -> Think -> Act -> Observe -> ...
```

Agent 的特点：

- 任务目标可以比较开放。
- 步骤不完全由开发者提前写死。
- 模型可以根据中间结果调整计划。
- 通常会使用工具，比如搜索、读文件、写代码、调用 API、运行测试。
- 需要状态管理、日志、失败恢复和停止条件。
- 能力上限更高，但不确定性和工程复杂度也更高。

例子：

- “阅读这个代码库，找出登录失败的原因并修复。”
- “调研某个主题，搜索资料，整理引用，生成报告。”
- “根据用户预算、日期和偏好规划旅行，并能反复调整。”
- “自动做代码审查，发现问题后生成测试并尝试修复。”

你需要重点理解：

- 目标定义：Agent 到底要完成什么。
- 环境：Agent 能看到什么，不能看到什么。
- 工具：Agent 可以采取哪些行动。
- 状态：当前任务进展、历史动作、观察结果如何保存。
- 计划：是否需要先拆解任务。
- 反馈：工具执行结果如何影响下一步。
- 停止条件：什么时候算完成，什么时候必须停下问用户。
- 安全边界：哪些操作不能让 Agent 自主执行。

适合使用 Agent 的场景：

- 任务步骤无法完全提前确定。
- 需要根据中间结果动态调整。
- 需要多工具协作。
- 任务需要探索、调试、搜索、反思。
- 用户更关心最终目标，而不是固定流程。

## 4. 三者对比

| 维度 | LLM App | Workflow | Agent |
|---|---|---|---|
| 核心目标 | 用模型增强单个功能 | 自动化固定流程 | 自主完成开放目标 |
| 控制权 | 主要在程序 | 主要在流程设计 | 部分交给模型 |
| 流程是否固定 | 基本固定 | 明确固定或有限分支 | 动态变化 |
| 是否需要工具 | 可选 | 常见但由流程控制 | 常见且由 Agent 选择 |
| 状态管理 | 简单 | 中等 | 复杂 |
| 可预测性 | 较高 | 高 | 较低 |
| 工程复杂度 | 低到中 | 中 | 中到高 |
| 适合初学者吗 | 最适合起步 | 适合进阶 | 需要谨慎设计 |

## 5. 关键判断问题

做项目时先问这几个问题：

1. 这个任务是不是一次 LLM 调用就能完成？
   - 是：优先做 LLM App。

2. 这个任务的步骤能不能提前写清楚？
   - 是：优先做 Workflow。

3. 这个任务是否需要根据中间结果不断调整？
   - 是：考虑 Agent。

4. 这个任务是否需要访问外部工具或环境？
   - 如果工具顺序固定，用 Workflow。
   - 如果工具选择依赖观察结果，用 Agent。

5. 出错成本高不高？
   - 高：优先 Workflow，加人工确认。
   - 低且需要探索：可以尝试 Agent。

## 6. 常见误区

误区 1：只要接了大模型就是 Agent。

不是。大多数总结、翻译、改写、分类应用只是 LLM App。

误区 2：Workflow 比 Agent 低级。

不是。很多生产系统更适合 Workflow，因为稳定、可控、好测试。

误区 3：Agent 可以完全不设计流程。

不对。专业的 Agent 开发不是放任模型自由发挥，而是设计目标、工具、状态、约束、停止条件和评估方式。

误区 4：Prompt 写好就够了。

不够。Prompt 只是入口，真正的工程还包括数据结构、工具接口、日志、测试、权限、安全和评估。

## 7. 对 AI Coding 的启发

对初学者来说，建议按这个顺序升级：

```text
LLM App -> Workflow -> Agent
```

不要一开始就追求“全自动智能体”。更专业的做法是：

- 先把一个功能做成稳定的 LLM App。
- 再把多个功能串成可控 Workflow。
- 最后只在确实需要动态规划和工具选择时，引入 Agent Loop。

在 AI Coding 中，Agent 最适合处理这类任务：

- 阅读代码库。
- 定位 bug。
- 修改文件。
- 运行测试。
- 根据错误继续修复。
- 总结变更。

这也是 Codex 这类 coding agent 的核心工作方式。

## 8. 本节小结

LLM App 是“模型能力组件”，Workflow 是“固定流程自动化”，Agent 是“目标驱动的动态行动系统”。

判断一个 AI 应用属于哪类，重点看三件事：

- 流程是否固定。
- 控制权在代码还是模型。
- 是否需要根据环境反馈动态决定下一步。

第一周学习时不要急着写复杂 Agent。先能清楚区分这三类系统，你后面学 ReAct、Plan-and-Solve、Reflection、RAG、MCP 时会轻松很多。
