# 第一章习题参考答案

说明：以下不是唯一标准答案，而是一份偏工程实践视角的参考答案。重点是训练你判断“什么算 Agent、什么时候用 Workflow、如何定义环境、如何设计反馈循环”。

## 习题 1：四个 case 是否属于智能体

### Case A：一台超级计算机

结论：单独看，不属于智能体。

理由：

- 它有强大的计算能力，但算力本身不等于智能体。
- 它没有被描述为拥有目标、感知环境、采取行动、根据反馈自主调整的能力。
- 如果这台超级计算机运行了一个自动交易系统、天气控制系统或智能调度系统，那么“运行在它上面的系统”可能是智能体，但硬件本身不是。

分类角度：

- 不是反射式智能体。
- 不是规划式智能体。
- 不是学习型智能体。
- 只是智能体可能使用的计算基础设施。

### Case B：特斯拉自动驾驶系统

结论：属于智能体。

理由：

- 它能通过摄像头、雷达、传感器等感知环境。
- 它有目标：安全驾驶、避免碰撞、遵守道路规则。
- 它能采取行动：刹车、变道、减速、保持车道。
- 它需要在动态环境中实时决策。

分类角度：

- 从反应速度看：强反应式智能体，因为障碍物出现时需要毫秒级响应。
- 从内部模型看：基于模型的智能体，因为它需要维护道路、车辆、障碍物、车道线等状态。
- 从目标看：基于目标的智能体，因为它要完成“安全到达目的地”。
- 从效用看：也带有基于效用的特征，因为它要权衡安全、舒适、速度、法规等多个目标。
- 从环境看：动态、部分可观察、随机、多智能体环境。

### Case C：AlphaGo

结论：属于智能体。

理由：

- 它能感知棋盘状态。
- 它有明确目标：赢得对局。
- 它能选择行动：落子。
- 它会评估未来局面并规划策略。

分类角度：

- 规划式智能体：会评估未来多步局面。
- 基于效用的智能体：会评估不同落子的胜率或价值。
- 学习型智能体：AlphaGo 的策略和价值判断来自训练。
- 环境相对特殊：棋盘完全可观察、规则确定、序贯、多智能体、竞争性环境。

### Case D：ChatGPT 扮演智能客服

结论：如果只是对话回答，不一定是完整智能体；如果能查询订单、分析原因、调用工具、给出方案并根据反馈调整，则属于 LLM Agent。

理由：

- 它能感知用户输入、订单信息、历史记录等上下文。
- 它有目标：解决投诉、遵守售后政策、安抚用户。
- 它能行动：查询订单、调用退款接口、生成补偿方案、转人工。
- 它需要处理模糊语言、情绪、政策约束和业务工具。

分类角度：

- LLM 驱动智能体。
- 混合式智能体：既需要快速回应用户情绪，也需要审慎判断政策和订单信息。
- 基于目标和效用的智能体：目标是解决问题，同时权衡用户满意度、公司成本、合规风险。
- 环境通常是部分可观察、动态、随机、多智能体环境。

## 习题 2：智能健身教练的 PEAS 模型

### Performance：性能度量

可以从以下指标评估智能健身教练是否做得好：

- 用户是否接近健身目标，比如减脂、增肌、提升耐力。
- 训练计划是否安全，是否避免过度训练和运动损伤。
- 实时指导是否准确，比如心率过高时及时降低强度。
- 动作纠正是否有效，比如减少错误姿势。
- 用户依从性和满意度，比如是否愿意持续训练。
- 饮食建议是否合理，是否符合用户目标、身体状况和偏好。
- 长期指标改善，比如体脂率、肌肉量、最大摄氧量、静息心率。

### Environment：环境

智能健身教练所处环境包括：

- 用户身体状态：心率、体温、疲劳程度、睡眠质量、伤病情况。
- 运动场景：健身房、家里、户外跑步、骑行、游泳等。
- 训练设备：可穿戴设备、手机、摄像头、智能手表、健身器械。
- 外部条件：天气、场地、时间、饮食记录、日程安排。
- 用户目标：减脂、增肌、提升耐力、康复训练。
- 用户反馈：太累、动作不适、疼痛、喜欢或不喜欢某类训练。

### Actuators：执行器

智能体可以采取的行动包括：

- 语音指导：提醒节奏、呼吸、姿势。
- 训练计划调整：增加或降低强度、更换动作、安排休息。
- 实时预警：心率过高、动作危险、疑似过度疲劳。
- 视觉反馈：在 App 中展示动作纠正提示。
- 饮食建议：推荐热量、蛋白质摄入、补水建议。
- 训练报告：总结训练效果和下一步计划。

### Sensors：传感器

智能体可获得的信息来源包括：

- 可穿戴设备：心率、步数、消耗热量、睡眠、血氧。
- 手机传感器：GPS、加速度计、陀螺仪。
- 摄像头：动作姿态、角度、稳定性。
- 用户输入：目标、体重、饮食、疲劳程度、疼痛反馈。
- 第三方数据：天气、食物数据库、运动记录平台。

### 环境特性分析

- 部分可观察：智能体无法完全知道用户真实疲劳、疼痛程度、肌肉恢复情况，只能通过设备和反馈推断。
- 随机性：同样训练计划在不同天效果不同，受睡眠、饮食、压力、天气影响。
- 动态性：用户状态在训练过程中实时变化，比如心率突然升高。
- 序贯性：今天的训练会影响明天的恢复和长期效果。
- 连续性：心率、速度、动作角度等是连续变量。
- 多目标性：安全、效果、体验、坚持度之间需要权衡。
- 个体差异强：同样计划对不同用户效果不同，需要个性化。

## 习题 3：电商退款 Workflow 与 Agent

### 方案 A：Workflow 的优点

- 规则清晰，容易解释。
- 稳定可控，适合高频标准化流程。
- 容易审计，便于合规和责任追踪。
- 成本较低，开发和维护相对简单。
- 对边界明确的退款场景很高效。

### 方案 A：Workflow 的缺点

- 对复杂情况不够灵活。
- 很难理解用户情绪、特殊背景、商品真实状况。
- 规则过多后会变得臃肿。
- 容易被用户钻规则漏洞。
- 对新业务、新商品、新政策适应慢。

### 方案 B：Agent 的优点

- 能理解更复杂的上下文，比如用户历史、投诉内容、商品状态。
- 能动态调用工具，比如查订单、物流、聊天记录、商品政策。
- 能给出更个性化的处理方案。
- 适合处理非标准、长尾、模糊案例。
- 可以兼顾用户体验，比如解释原因、安抚情绪、提出替代补偿。

### 方案 B：Agent 的缺点

- 可预测性较差。
- 可能产生错误判断或幻觉。
- 审计和责任归因更难。
- 需要严格权限控制，不能让 Agent 随意退款。
- 成本更高，包括模型调用、工具集成、评估和监控。

### 什么时候 Workflow 更合适

- 规则明确、流程稳定。
- 处理量大，单笔价值较低。
- 合规要求高，需要可解释。
- 决策边界清楚，比如 7 天无理由、金额阈值、商品类型。
- 不能接受模型自由裁量。

### 什么时候 Agent 更有优势

- 案例复杂，信息分散在多个系统中。
- 需要理解自然语言投诉、情绪和上下文。
- 需要综合用户历史行为、商品状态、物流异常等信息。
- 需要生成解释、协商方案或补偿建议。
- 需要辅助人工客服，而不是完全替代人工审批。

### 如果我是负责人，会怎么选

我不会直接采用纯 Agent 自动决策退款。更稳妥的做法是：标准场景用 Workflow 自动处理，复杂场景让 Agent 辅助人工或给出建议。

原因：

- 退款涉及真实资金和合规风险。
- Workflow 能覆盖大量标准场景。
- Agent 适合处理边界模糊的复杂场景，但需要人工兜底。

### 方案 C：Workflow + Agent 混合方案

推荐方案：

```text
退款申请
-> Workflow 做基础规则判断
-> 标准低风险案例自动通过或拒绝
-> 中高风险或复杂案例进入 Agent 分析
-> Agent 查询订单、物流、用户历史、商品政策
-> Agent 给出建议和理由
-> 人工客服或主管确认
-> 结果写入日志，用于后续评估
```

可以加入安全边界：

- 金额低于某阈值且规则明确，自动处理。
- 高金额、特殊商品、异常用户，必须人工确认。
- Agent 只能建议，不能直接执行高风险退款。
- 所有工具调用和判断理由都记录日志。

## 习题 4：给智能旅行助手添加记忆、备选和反思

### 功能 1：记住用户偏好

设计思路：

- 增加 Memory 模块，保存用户偏好。
- 偏好可以包括：预算、景点类型、酒店偏好、饮食禁忌、交通方式、旅行节奏。
- 每次对话后，从用户输入中抽取偏好并更新记忆。
- 推荐前先检索相关记忆，把偏好加入上下文。

简单流程：

```text
用户输入
-> 提取偏好
-> 更新 Memory
-> 检索相关偏好
-> 生成推荐
```

示例：

用户说：“我不太喜欢网红景点，更喜欢历史文化类的地方，预算别太高。”

记忆可以保存为：

```json
{
  "attraction_preference": ["history", "culture"],
  "avoid": ["internet-famous spots"],
  "budget": "medium-low"
}
```

### 功能 2：门票售罄时自动推荐备选方案

设计思路：

- 给景点查询或预订工具增加状态返回，比如 `available: false`。
- 当 Observation 显示门票售罄时，不直接失败，而是触发替代推荐策略。
- 替代策略可以按相似主题、距离、价格、开放时间排序。

Thought-Action-Observation 可以变成：

```text
Thought: 用户想去历史文化类景点，我需要先查门票。
Action: check_ticket("故宫")
Observation: 门票已售罄
Thought: 原景点不可用，需要寻找相同类型、同区域或同预算的备选景点。
Action: search_alternative_attractions(type="history", location="北京", date="...")
Observation: 返回国家博物馆、孔庙和国子监、景山公园
Thought: 选择最符合用户偏好的备选方案并解释原因。
```

### 功能 3：用户连续拒绝 3 个推荐后反思并调整策略

设计思路：

- 增加拒绝计数器和拒绝原因记录。
- 当连续拒绝次数达到 3 时，触发 Reflection。
- Reflection 分析：我是否误解了用户偏好？预算是否不匹配？推荐是否太热门、太远、太贵？
- 然后向用户澄清或调整推荐策略。

流程：

```text
推荐方案
-> 用户拒绝
-> 记录拒绝原因
-> reject_count += 1
-> 如果 reject_count >= 3
   -> Reflection
   -> 总结可能误判
   -> 询问澄清问题或改变推荐方向
```

示例反思：

```text
我连续推荐了热门历史景点，但用户都拒绝了。
可能原因：用户不喜欢人多景点，或者想要更小众、更轻松的路线。
下一步：推荐小众文化街区、博物馆夜场或低强度路线，并询问用户是否偏好安静体验。
```

## 习题 5：系统 1 与系统 2 的智能体应用

这里选择“医疗问诊助手”作为场景。

目标：辅助用户初步描述症状、识别风险、整理信息，并在必要时建议线下就医。注意它不能替代医生诊断。

### 系统 1：快速直觉处理的任务

适合由系统 1 处理的任务：

- 快速理解用户自然语言描述。
- 识别常见症状关键词，比如发热、咳嗽、胸痛、头晕。
- 判断是否出现高危信号，比如呼吸困难、意识模糊、剧烈胸痛。
- 将用户输入初步分类，比如感冒类、消化类、皮肤类、外伤类。
- 生成温和、易懂的追问。

系统 1 的价值是快，适合做初筛、模式识别和语言理解。

### 系统 2：慢速推理处理的任务

适合由系统 2 处理的任务：

- 按医学规则或指南做结构化推理。
- 检查症状之间是否存在矛盾。
- 根据年龄、病史、用药、持续时间评估风险。
- 决定是否需要紧急就医。
- 生成可解释的建议。
- 检查回答是否越界，比如是否给出了不该给的诊断或处方。

系统 2 的价值是谨慎，适合高风险判断、规则校验和解释。

### 两个系统如何协同

可以设计为：

```text
用户描述症状
-> 系统 1 快速抽取症状、情绪和关键词
-> 系统 1 初步分类并生成追问
-> 收集更多信息
-> 系统 2 根据结构化信息做风险评估
-> 系统 2 检查医学安全边界
-> 输出建议：自我观察、预约医生、立即就医
```

关键原则：

- 系统 1 负责快和广。
- 系统 2 负责慢和稳。
- 高风险决策必须经过系统 2。
- 最终输出要有边界，不能伪装成医生确诊。

## 习题 6：幻觉、循环限制与智能评估

### 1. 为什么 Agent 会产生幻觉

原因包括：

- LLM 的生成目标是预测合理文本，不是天然保证事实正确。
- 上下文信息不足时，模型可能用训练中的常见模式补全。
- 工具结果没有正确接入，模型可能凭记忆回答。
- Prompt 约束不清，模型不知道必须引用来源或承认不知道。
- 多步推理中前一步错误会传递到后续步骤。
- Agent 对工具返回结果理解错误，导致错误行动。
- 记忆或 RAG 检索到了错误、过时、不相关的信息。

降低幻觉的方法：

- 关键事实必须通过工具或检索验证。
- 要求引用来源。
- 允许模型回答“不知道”。
- 对高风险输出加规则校验或人工审核。
- 记录中间步骤，方便追踪错误来源。

### 2. 如果没有最大循环次数，Agent 可能陷入什么问题

可能出现：

- 无限循环：反复调用同一个工具，得不到新信息。
- 目标漂移：执行过程中偏离原始任务。
- 过度探索：不断搜索更多信息，但不收敛。
- 成本失控：模型调用次数、工具调用次数、时间成本持续增加。
- 状态膨胀：上下文越来越长，反而降低判断质量。
- 反复自我修正：Reflection 一直觉得还可以改，永远不结束。
- 错误放大：基于错误观察继续行动，越走越偏。

所以 Agent 必须有停止条件，例如：

- 最大循环次数。
- 最大花费或最大时间。
- 达到明确完成条件。
- 连续多次无进展时停止。
- 高风险操作前暂停并询问用户。
- 工具重复失败时停止。

### 3. 如何评估 Agent 的智能程度

准确率重要，但不够。

因为 Agent 不只是回答问题，它还要规划、调用工具、处理异常、控制成本、解释结果。更完整的评估指标包括：

- 任务完成率：最终是否完成用户目标。
- 工具调用准确率：是否选择了正确工具，参数是否正确。
- 规划质量：步骤是否合理，是否能根据反馈调整。
- 事实正确性：关键事实是否可靠。
- 鲁棒性：工具失败、信息缺失、用户改变需求时能否恢复。
- 安全性：是否避免危险操作、越权操作、泄露隐私。
- 可解释性：是否能说明为什么这样做。
- 成本：耗时、调用次数、token 成本。
- 用户满意度：结果是否真正有用。
- 可复现性：同类任务表现是否稳定。

一个更专业的评估方式：

```text
准备一组真实任务
-> 规定成功标准
-> 记录 Agent 的每一步行动
-> 统计成功率、错误类型、成本和耗时
-> 人工抽查关键样本
-> 根据失败原因改进工具、Prompt、流程或评估集
```

## 总结

第一章习题的核心不是记概念，而是建立三个判断能力：

1. 判断一个系统是否是真正的智能体。
2. 判断一个任务更适合 Workflow 还是 Agent。
3. 判断 Agent 设计中哪些部分需要目标、工具、记忆、反馈、停止条件和评估。

如果你能用这些角度分析一个具体项目，就已经开始从“会和 AI 对话”进入“会设计 AI 系统”了。
